# Frida 探针库指导书（给 Codex）

> 用途：目标 APK 取证。目标只有一个——**从样本里逼出可追溯的归属线索**（真实源站域名/IP、后端端点、租户/企业号、解密 key、聊天/支付物证）。
> 性质：本项目为**个人安全研究 / 测试**用途；分析中如发现可疑线索，可提供给授权方处置。文中"归属分析 / 溯源"等仅说明合法分析方向，不代表任何官方身份。
> 本书只讲**怎么用、怎么选、怎么扩、红线**；探针源码在同目录 `probe-templates/`。所有探针都是**单文件自包含纯 frida JS，可直接 `-l` 注入**，每个 hook 各自 `try/catch`、抓不到打"未命中+下一步换探针"、绝不静默、绝不假装抓到。
> 共 **46 个探针**。在 23 个基础上新增：Telegram/MTProto 改包专线（telegram-mtproto / activity-nav 绕加载页+视频门控 / netstat）、冷启动锚点（sdk-appkey / objstore-config）、资金链·C2·短信转发 P0（pay-sdk / push-c2-inbound / sms-forward-outbound）、新协议栈/RTC（cronet-quic-http3 / rn-bridge-native / mqtt-xmpp-im / protobuf-grpc / rtc-join）、凭据深化/原生定位（native-crypto-key / keystore-alias-tracer / mmkv-realm-wcdb-key / register-natives）、取证·毁证·数据外泄·远控（evidence-wipe-interceptor / self-uninstall-guard / sensitive-data-access / accessibility-abuse / nfc-hce-relay / multiopen-virtualapp-detect）。

## 自助更新（Codex 每次开工先做）

探针库、本指导书、fxapk 同在仓库 `s-silt/fxapk`，**一条 `git pull` 全部更新**，不再等手递压缩包：

```
cd <fxapk 仓库目录> && git pull            # 拿到最新 fxapk CLI + 本指导书 + probe-templates/
pip install -e . --upgrade                 # 让 fxapk 命令同步到最新（含 probe-leads 等）
```

- 探针在本目录 `probe-templates/*.js`；本指导书 `docs/codex/frida-probes/指导书.md`。
- 拿到样本**先查本库**（§2 决策表按症状选探针），有现成的别手写；确实没有再按 §5 扩。
- 探针抓到的线索用 **`fxapk probe-leads`** 聚成溯源台账并回灌 report.json（见 §4「探针输出 → 溯源台账」）。

## 1. 开篇决策树：无账号能否触达真实源站？

这类目标样本的典型结构是**两层后端**：

```
        [无账号冷启动可见]                      [企业号 gating 后才下发]
   ┌──────────────────────────┐        ┌──────────────────────────────────┐
   │  CDN / OSS 配置分发层      │  ──►   │  真实业务源站                     │
   │  (拉 config.json / 引导)   │ 企业号 │  客服后台 / 聊天网关 / 支付通道    │
   └──────────────────────────┘  gating └──────────────────────────────────┘
```

**核心痛点**：真源站（客服/聊天/支付）常常只在"按企业号(如 flyno5)下发的配置响应"里。没有效企业号 → 服务器不下发 → 探针只看到 CDN/OSS 分发层。所以策略是**在登录前/无账号阶段把能拿的全拿到，再用不靠登录的手段把真源站逼出来**。

### 决策树（每步指到具体探针）

**Step 1 — 无账号冷启动全量抓（不输任何企业号，纯冷启动）**
能拿到的：配置分发层 URL/响应、内嵌默认配置、SNI/TLS 握手目标、DNS 解析目标、首屏拉取的所有端点结构。
拿不到的：企业号 gating 之后才下发的真源站。
- 主力：`coldstart-config-hook.js`（OkHttp execute + InetAddress.getAllByName + URL.openConnection + AssetManager.open + SharedPreferences 首读 → 一网打尽冷启动期所有出站结构、DNS 目标、内嵌资产配置）
- 补强：`http-url-hook.js`（出站 URL/头/明文体）、`dns-hook.js`（域名→IP 解析记录，直接归属线索）、`native-ssl-hook.js`（拿不到明文流量时的 SNI/TLS 兜底）

**Step 2 — 找配置端点 + 枚举企业号**
冷启动里 `coldstart-config-hook.js` 会用启发式标 `[TENANT?]`（疑似企业号字段）和 `[BACKEND!]`（疑似真后端域名）。
- 用 `coldstart-config-hook.js` 锁定**配置端点**（接收企业号、下发配置的那个 URL）的完整结构（路径、签名字段、时间戳）。
- 拿到案卷里**已知企业号**后，用 `tenant-enum-helper.js` 小批量重放该配置端点，观察"哪个企业号能让服务器下发真源站配置"。
  - 红线见 §6：主动枚举/重放是主动外联，必须先确认授权，`TENANT_LIST` 默认空。

**Step 3 — 配置/响应是密文 → 解密内嵌配置**
很多样本把下发配置或内嵌默认配置加密。
- `cipher-hook.js`（Cipher.doFinal/update 明文 + AES key/iv + opmode 方向）→ 拿到解密后的配置明文，里面常直接含真源站域名。
- 配置端点带**签名/时间戳**导致 `tenant-enum-helper.js` 被服务器拒 → 先用 `cipher-hook.js` / `sign-hook.js` 反推签名构造，再重放。
- 内嵌配置走 Base64/JSON 信封 → `encoding-hook.js`（Base64/Gson/JSONObject/URLEncoder → 加密前的明文信封契约）。

**Step 4 — 溯源穿透 CDN/OSS**
即使只拿到分发层，也能溯源：
- `dns-hook.js` 的域名→IP 记录、`coldstart-config-hook.js` 的 OSS bucket/CDN 回源域名 → 走 CDN/OSS 服务商调取**回源真实 IP、bucket 归属、上传者**。
- `[BACKEND!]` 标注的疑似真后端域名 → 直接落地为域名/IP 归属线索。

**Step 5 — 真源站只在登录后/功能里出现（gating 攻不动）**
- 若有合法测试账号或案卷企业号：spawn 注入全套，进客服/进会话/走支付，让 `websocket-hook.js`（聊天帧）、`okhttp-interceptor-hook.js`（请求/响应全链）、`webview-hook.js` + `webview-inject-js.js`（H5 端点 + 页面内 fetch/WS/XHR）把真源站抓出来。
- 完全无账号且 gating 死守：以 Step 1–4 拿到的分发层 + DNS + 解密配置 + CDN/OSS 溯源作为线索闭环，不靠登录。

## 2. 症状 → 选哪个探针（决策表）

| 症状 | 含义 | 选探针 |
|---|---|---|
| ① HTTPS 密文体（流量有包但 body 是密文） | 应用层加密 | `cipher-hook.js`（明文+key/iv）；信封结构 `encoding-hook.js`；签名字段 `sign-hook.js` |
| ② 聊天内容不在 HTTP 流量里 | 走 WebSocket | `websocket-hook.js`；若是 native 帧 → `native-ssl-hook.js` |
| ③ 端点点某功能才出现 | UI 触发 | spawn 注入后手动触发功能，配 `okhttp-interceptor-hook.js` / `http-url-hook.js` 全量抓 |
| ④ 没流量 / 秒退 | pinning / 反 frida / 非 OkHttp / native 发包 | 先 `anti-detection-hook.js` + `ssl-unpinning-hook.js`（最先注入）；仍无流量见下 |
| ⑤ 秒退（启动即崩/退） | 反 frida / 反调试 / root 检测（Java + **native** 层） | `anti-detection-hook.js`（Java层，最先注入）；仍秒退 → 叠加 `anti-detection-native.js`（native /proc/maps、TracerPid、27042 端口） |
| ⑥ 裸 socket（绕过 OkHttp 直发） | 自实现网络栈 | `socket-hook.js`（send/recv 字节流）；全空 → `native-ssl-hook.js`（native 层发包） |
| ⑦ H5 壳（界面是网页） | WebView 承载业务 | `webview-hook.js`（端点/JS 桥）；H5 内真实 fetch/WS/XHR → `webview-inject-js.js`（注入页面 JS 回吐） |
| ⑧ 配置 gating（无企业号不下发真源站） | 租户 gating | `coldstart-config-hook.js`（配置端点+`[TENANT?]`）→ `tenant-enum-helper.js`（授权后枚举重放） |
| ⑨ SQLCipher / 本地库加密 | 聊天/凭据落加密库 | `sqlite-hook.js`（execSQL/rawQuery 明文 + openOrCreate/changePassword 的 key，含 char[]/byte[] 解码）|
| ⑩ 签名密文（请求带 sign/sig 字段） | 防篡改签名 | `sign-hook.js`（被签明文+算法+HMAC key → 离线自造签名）|
| ⑪ 凭据/租户ID 在本地存储 | SP/MMKV 持久化 | `sharedprefs-hook.js`（SharedPreferences + MMKV 读写明文，含租户ID/token）|
| ⑫ DNS 解析目标存疑 | 域名→IP 溯源 | `dns-hook.js`（getAllByName/OkHttp Dns → 解析记录）|
| ⑬ OkHttp 有但看不到响应体 | 拦截器层 | `okhttp-interceptor-hook.js`（Interceptor 链 + Retrofit.create 请求/响应全文）|
| ⑭ 静态看不到真实逻辑（壳/加固/二次释放） | 运行时才放真实 dex | `dexload-hook.js`（dump 动态加载的 dex，adb pull 回灌静态）|
| ⑮ app 跑外部命令 / 拉起子进程 | 运营端动作 / 二次 payload | `process-exec-hook.js`（Runtime.exec / ProcessBuilder / libc system/execve）|
| ⑯ 不知哪个 .so 承载 BoringSSL/加密 | native 模块定位 | `native-loadlib-hook.js`（列加载的 .so）→ 把模块名喂 `native-ssl-hook.js` |
| ⑰ TLS 抓到密文但解不开（Flutter/QUIC/非标准栈/pin 死代理） | TLS 主密钥未导出 | `tls-keylog.js`（导 SSLKEYLOGFILE + 另抓 pcap → Wireshark 离线解**任意栈**）；或 `native-ssl-hook.js` 直接 hook 明文 |
| ⑱ 加固壳/二代壳，dexload 抓不到真实 dex | 壳 mmap 自解密、不走标准 ClassLoader | `memdex-dump.js`（内存扫 dex magic 全量 dump，adb pull 回灌静态） |
| ⑲ **Telegram/XIN 二开改包，登录/聊天无明文、普通抓包 `endpoint_total=0`** | 走自建 MTProto 协议，HTTP/OkHttp 代理看不见 | `telegram-mtproto-hook.js`（TLRPC 对象层抓登录明文+聊天字段+DC 接入节点；不破 MTProto 加密）。类挂不上（加固未释放）→ 先 `memdex-dump.js`/`dexload-hook.js` 放真实 DEX，再 REPL 调 `fxTgScan()` 看真实类名回填 `CAND_*` |
| ⑳ **卡「加载页→视频→登录页」多级门控，到不了登录页抓包** | splash/引导视频挡住登录链路（视频是"播完才进登录"型），后续探针够不着 | `activity-nav-hook.js`：**视频层**默认 `SKIP_VIDEO` 自动 `seekTo` 末尾触发真实播完→跑 app 自己的进登录逻辑（对"跳 Activity"和"单 Activity 内切 View/Fragment"如 `LoginNewActivity$PasswordView` 都有效）；不灵→REPL `fxEndVideo()` 手动触发完成回调、`fxFragments()` 看单页内登录 View。**强跳**→`fxGoto('登录类名')`（先 `fxListActivities()` 看 `[LEAD-> 登录页]`）。**一键**→`fxSkipGate('登录类名')`。倒计时型→`SKIP_COUNTDOWN`，最猛→`KILL_SPLASH`（先观察防打断导航）。到登录页后让 telegram-mtproto/cipher/http 接住明文 |
| ㉑ **native 协议接入节点（IP:port）抓不到、`/proc/net/tcp` 只见 SYN_SENT** | 自建协议在 native 层 connect，Java 看不到对端 | `netstat-hook.js`（/proc/net/tcp(6) 周期采样 + libc connect 实时，含 SYN_SENT/ESTABLISHED 状态、UID 过滤）；TLRPC 场景 `telegram-mtproto-hook.js` 自带 connect 兜底；五元组/SNI 补 `native-ssl-hook.js` |
| ㉒ **要找 appKey / 渠道 / 租户标识（安装归因·统计·推送 SDK）** | OpenInstall/友盟/Firebase/个推/极光的 init 传参里藏唯一租户标识 | `sdk-appkey-hook.js`（**必须 `-f` spawn**，init 在 Application 极早期）。类名混淆抓不到 → 退 `sharedprefs-hook.js`/`sqlite-hook.js` 看落盘 appKey，或 apktool 解 manifest `<meta-data>` |
| ㉓ **冷启动从多云对象存储分散下发配置（OSS/BOS/ZOS/COS）** | 配置/资源走对象存储分发层（抗封堵），真后端写在下发正文里 | `objstore-config-hook.js`（**`-f` spawn**，抓 bucket 锚点 + 配置正文 + `[BACKEND]` 抽真后端）。正文是密文 → 配 `cipher-hook.js`/`encoding-hook.js` 还原；走 native/Flutter 栈 → 配 `native-ssl-hook.js`+`socket-hook.js` |
| ㉔ **支付/收款相关**：查资金归集账户、商户号、收款码、notify_url | 第三方支付 SDK 调起 | `pay-sdk-hook.js`（**`-f` spawn**，支付宝 payV2 的 seller_id/notify_url、微信 PayReq.partnerId 商户号、银联 tn）。走 H5 收银台/聚合第四方→回退 `webview-hook.js`/`http-url-hook.js` 抓创建订单接口 |
| ㉕ **找 C2 指令 / 真后端域名 / 推送 regId**：app 平时只连 CDN、装死等服务器下令 | C2 走推送通道下发 | `push-c2-inbound-hook.js`（**`-f` spawn**，FCM/个推/极光收消息回调 payload→真 C2 域名+`cmd=*` 指令物证；regId/clientid 调实名）。抓不到→`websocket-hook.js`（长连下发）/`http-url-hook.js`（轮询拉指令） |
| ㉖ **短信验证码转发 / 卡农**：怀疑收到 OTP 即转发、接管验证码 | 出站短信转发 | `sms-forward-outbound-hook.js`（`SmsManager.send*` 的 destinationAddress=上家/接码号→运营商实名；OTP 成对落盘=倒卖物证）。走网络上报而非短信→回退 `http-url-hook.js`/`socket-hook.js` |
| ㉗ 抓不到 OkHttp/普通 HTTP，流量是**加密 UDP/QUIC(443)**；界面非 H5 | Chromium Cronet(QUIC/HTTP3) | `cronet-quic-http3-hook.js`（钩 Builder + Callback 真实子类）。纯 native libcronet→`native-ssl`/`socket` |
| ㉘ 界面是 **React Native**（含 `com.facebook.react`、JS bundle）| RN bridge | `rn-bridge-native-hook.js`。RN0.74+ Bridgeless→回退 `native-ssl`/`socket` |
| ㉙ 走 **MQTT/XMPP** 私有 IM（Paho/HiveMQ）| MQTT/XMPP | `mqtt-xmpp-im-hook.js`（broker+user/pass+topic）|
| ㉚ body 是 **protobuf/gRPC**（二进制、`grpc-` 头）| protobuf | `protobuf-grpc-hook.js`（边界自吐+tag/wire-type）|
| ㉛ **音视频会话/视频认证**（声网/腾讯TRTC/即构 RTC）| 音视频入会 | `rtc-join-hook.js`（appId+channel/room）。媒体节点 IP→配 `netstat` |
| ㉜ 流量/库解不开，**key 在 native libcrypto** | native 对称加密 | `native-crypto-key-hook.js`（EVP/AES key+iv）|
| ㉝ key 在 **AndroidKeyStore 托管**，要判能否脱机解 | 托管密钥 | `keystore-alias-tracer-hook.js`（securityLevel 裁决）+ 软件级配 `cipher-hook` |
| ㉞ 加密库**不是 SQLCipher**（MMKV/Realm/WCDB）| 非标加密库 | `mmkv-realm-wcdb-key-hook.js`（cryptKey/encryptionKey/sqlite3_key 脱机解库）|
| ㉟ 加固样本 **native 无符号入口**要定位 | RegisterNatives | `register-natives-hook.js`（类#方法→so!offset，喂 native-ssl/crypto-key）|
| ㊱ 样本**自毁/删聊天转账记录** | 毁证 | `evidence-wipe-interceptor-hook.js`（删前抢救+记调用）+ `self-uninstall-guard-hook.js`（wipeData/自卸载告警）|
| ㊲ 偷读**通讯录/短信/位置/剪贴板/IMEI/录屏** | 数据收割 | `sensitive-data-access-hook.js`（固证读了什么；剪贴板=收款码劫持）|
| ㊳ **无障碍**自动点击/读屏（远控转账/抓验证码）| 无障碍滥用 | `accessibility-abuse-hook.js`（dispatchGesture/performAction/onAccessibilityEvent）|
| ㊴ **NFC 中继盗刷**（Ghost Tap/NGate）| HCE 中继 | `nfc-hce-relay-hook.js`（IsoDep/HCE APDU，SELECT AID→发卡行）|

## 4. 怎么跑 / 怎么组合

### spawn vs attach
- **冷启动取证必须 spawn**（`-f <包名>`）：配置拉取、内嵌配置解密、首屏 DNS/TLS、动态 dex 释放都发生在启动瞬间，attach 上去时早已错过。
  ```
  frida -U -f com.target.app -l anti-detection-hook.js -l ssl-unpinning-hook.js -l coldstart-config-hook.js
  ```
- attach（`-U <包名>` 或 PID）只用于：进程已起、要补抓某个**点功能才触发**的端点（症状③）。

### 注入顺序（关键）
**反检测/解 pinning 必须最先注入**，否则业务探针还没挂上就被反 frida 弄退、或 HTTPS 仍是密文：
```
1) anti-detection-hook.js     # 先绕检测，防秒退（Java 层）
   anti-detection-native.js   #   仍秒退时一起上（native 层 /proc/maps、TracerPid、27042）
2) ssl-unpinning-hook.js      # 再解 pinning，放明文
2.5) memdex-dump.js / dexload-hook.js   # 【改包加固必备】先放运行期释放的真实 DEX，否则 telegram-mtproto/sdk 的类挂不上
3) <业务探针，按需多个 -l 叠加>  # coldstart / http / cipher / websocket / tls-keylog ...
```
`-l` 可叠加任意多个，**按上面顺序排列**（前置在前，业务在后）。多探针同时注入彼此独立、各自 try/catch，不互相影响。

**Telegram/MTProto 改包专线的组合（本轮新增探针）：**
```
3a) sdk-appkey-hook.js / objstore-config-hook.js   # 都是冷启动极早期触发，必须 -f spawn 在场（抓 appKey、对象存储 bucket+配置正文）
3b) telegram-mtproto-hook.js                        # TLRPC 层登录/聊天/接入节点；配 native-ssl(五元组/SNI) 与 netstat(全连接表) 交叉验证 :30113 是否真建连
3c) activity-nav-hook.js                            # 卡「加载页→视频→登录」才上：SKIP_VIDEO 默认自动 seekTo 末尾跳过视频；不行 REPL fxEndVideo()/fxSkipGate('登录类名')
```
- **activity-nav 是"卡住才上"的现场工具**，不必每次挂。先 spawn 跑 3a+3b 观察冷启动；发现停在 splash/视频、登录页 `onCreate` 一直不出现 → 再叠加 activity-nav，等它 3s 后打印 `[acts]` 全表，挑 `[LEAD-> 登录页]` 类名，`fxSkipGate('类名')` 一键过门控。
- **telegram-mtproto 的类挂不上**（加固运行期才释放 14 DEX）→ 先靠 2.5 的 memdex-dump 放真实 DEX，再 REPL `fxTgScan()` 看真实类名回填。

**资金链/C2/短信转发 P0 组合（只读取证仪，按需挂）：**
```
3a) pay-sdk-hook.js / push-c2-inbound-hook.js / sms-forward-outbound-hook.js   # 均 -f spawn（支付调起/push 注册/短信转发都可能冷启动早期触发，attach 会漏首帧）
```
- 三者都靠 spawn 抢在首次触发前就位；可单挂或组合，彼此独立。
- **push-c2 需样本完成推送注册后再触发**（或反复运行），否则 `enumerateLoadedClasses` 找不到 `*MessagingService` 子类；FCM 需带 GMS 真机。
- **三者都是只读检测仪**：仅 hook+打印样本"对外做了什么"，不发送/不拦截/不修改（支付/短信/推送原调用一律透传放行）。
- 抓不到回退：pay→`webview-hook`/`http-url`；push-c2→`websocket-hook`/`http-url`；sms→`http-url`/`socket-hook`。

### 输出落文件
frida 控制台输出量大，务必落盘留存（取证可追溯）：
```
frida -U -f com.target.app \
  -l anti-detection-hook.js -l ssl-unpinning-hook.js -l coldstart-config-hook.js \
  -o capture_$(date +%Y%m%d_%H%M%S).log
```
（`-o` 写日志文件；或用 `frida ... | tee` 留存。）

### 触发功能
- 无账号路线：spawn 后**静置等冷启动跑完**，看 `[TENANT?]`/`[BACKEND!]`/DNS/配置端点。
- 有账号/案卷企业号路线：spawn 后**依次手动触发**首屏→拉配置→进客服→进会话→走支付，让对应探针（websocket/interceptor/webview/webview-inject/sqlite）抓到真源站与物证。

### 探针输出 → 溯源台账（`fxapk probe-leads`）
探针往 console 吐的 `[tag][LEAD-...]` 散点，**别人肉眼捞**——落盘后用 fxapk 一条命令聚成台账并回灌报告：

```
frida -U -f <包名> -l anti-detection-hook.js -l ssl-unpinning-hook.js -l <业务探针...> -o probe.log   # 抓取落盘
fxapk probe-leads probe.log                          # 打印溯源台账（按归属类别分组 + 溯源落点 + 取证完备性诊断）
fxapk probe-leads probe.log --md 台账.md --json 台账.json   # 落盘台账
fxapk probe-leads probe.log --into report.json       # 直接把探针线索回灌进 fxapk 报告的 leads（去重）
```

- 台账按 `LeadCategory` 分组（PAYMENT/SMS_FORWARDING/SELF_HOSTED_IM/VICTIM_DATA/REMOTE_CONTROL/CRYPTO_RECIPE/DOMAIN/IP…），每组给 **where_to_request**（向谁溯源、要什么）。
- 末尾「**取证完备性**」诊断三类溯源价值轴（**定人 / 穿透 / 固证**）哪类已覆盖、哪类没抓到→**该补跑哪个探针**——按它提示回头补抓，直到三轴齐。
- `--into report.json` 后，探针线索与 fxapk 静态/动态线索同构（`source=runtime-probe`、`is_runtime_seen=True`），一起进报告渲染、跨样本关联、套打溯源函件（`fxapk letters`）。

### 带外抓包 → 溯源台账（`fxapk pcap-leads`）—— frida 上不去/解不开时的另一条路
反 frida 秒退、TLS pinning、MTProto 等自建协议（`endpoint_total=0`）让探针够不着时，**不碰 App 本体**抓一份带外 pcap，照样能拿到**真实接入节点 IP:port + TLS SNI + DNS**——这就是穿透真源站、向云厂商调主机租户的归属锚点（**解不开密文也能追溯**）：

```
# 带外抓 pcap：① 网关/旁路由 tcpdump；② 设备端 PCAPdroid(免 root VpnService) 导出 .pcap；③ Wireshark .pcapng
fxapk pcap-leads capture.pcap                     # 台账：接入节点 IP:port + SNI + DNS（按 IP/DOMAIN 分组 + 溯源落点）
fxapk pcap-leads capture.pcap --into report.json  # 回灌进 report.leads（source=runtime-pcap，去重）
```

- 纯标准库解析（不需装 dpkt/scapy/tshark），支持经典 pcap / pcapng，Ethernet/RAW/Linux-SLL，IPv4/6，TCP/UDP，TLS ClientHello 的 SNI+JA3、DNS。
- 公网接入节点 IP→`IP` 线索（穿透）；SNI/DNS 域名→`DOMAIN` 线索（经 infra 分级，CDN/SDK 自动判无需溯源）；私网 IP 自动过滤。
- **和探针互补**：探针拿明文/落地数据（定人+固证），pcap 拿接入节点 IP/SNI（穿透）——两边 `--into` 同一 report.json，线索合并关联。

## 5. 怎么扩 / 改 hook 点

### house style（新探针必须遵守）
- 单文件自包含**纯 frida JS**，无外部依赖，可单独 `-l` 注入。
- 文件头 4 行中文注释：用途 / 适用 / 怎么跑 / 怎么改。
- Java 段用 `Java.perform` 包裹；native 段 `'use strict'` + `Interceptor`/`Module`。
- **每个 hook 各自 `try/catch`**，失败打 `[tag] xxx skip: <e>`，**绝不静默吞**。
- 标签前缀（`[http]`/`[cipher]`/`[coldstart]`…）+ 命中线索打 `[LEAD->后端]` / `[LEAD->凭据]` 等。
- 明文同时给**文本 + hex + base64**；二进制不盲 UTF-8（printable<85% 走 hex+b64）。
- 抓不到 → 打"未发现 + 下一步换哪个 hook 点/哪个探针"，**不假装抓到**。
- 每个 hook 写清"抓到什么 → 成什么归属线索"。

### 不命中怎么换栈（常见跨版本坑，已在各探针 notes/注释标注）
- **OkHttp 3.x/4.x**：RealCall 在 `okhttp3.RealCall`(3.x) vs `okhttp3.internal.connection.RealCall`(4.x)；RequestBody.create 参数顺序反转。→ 候选类名都试、两序兜底（见 coldstart/tenant-enum）。
- **重载歧义**（`has more than one overload`）：优先 `.overload(显式签名)`，取不到退唯一重载、再退 `.implementation`，外层 try 兜底（见 coldstart 的 `_hookMethod`）。
- **okhttp3.Dns 内部类名**跨版本极不稳（`Dns$1`/匿名类）→ 遍历候选类名逐个 try，最终都走 `InetAddress.getAllByName` 兜住（见 dns-hook）。
- **registerClass 重复注册**（`already registered`）→ 顶层只注册一次后复用（见 ssl-unpinning / okhttp-interceptor）。
- **char[]/byte[] 密码**：直接 `+` 拼接得 `[C@`/`[B@` 垃圾 → 按类型逐元素解码（byte 负值 +256），见 sqlite-hook 的 `decodePassword()`。
- **frida 无 btoa/atob**：用纯 JS 查表 base64，别用 `btoa()`（必崩，见 socket-hook）。
- **setTimeout 回调调 Java 方法**：回调线程未 attach VM → 必崩 → tick 体整体包 `Java.perform`（见 tenant-enum）。
- **MMKV 写**：`encodeString` 是 private native，公开写是 `encode/putString`（见 sharedprefs）。
- **ResponseBody.string()** 无参（按 Content-Type 推断），传 Charset 必抛 → 用无参 `peek.string()`（见 okhttp-interceptor）。
- **WebView 注入回吐**：app 子类覆盖 `onConsoleMessage` 不调 super 时 `[H5]` 回吐不出来 → 看 `webview-inject-js` 打印的 `setWebChromeClient` 实际类名，改 hook 那个子类。
- **InMemoryDexClassLoader**：dex 在内存 ByteBuffer，必须落盘才拿得到（见 dexload-hook 的 `dumpBuffer`）。
- 拿不到明文/native 发包：业务探针全空 → 转 `native-ssl-hook.js`（libssl/libflutter）或 `socket-hook.js`；不知哪个 .so → 先 `native-loadlib-hook.js`。

### 回灌线索（重要）
现场跑出的 `[LEAD->*]` 线索（真源站域名、解库 key、租户ID、签名构造）**手动并入 report / track manual**——人工录入溯源台账，**不是** `--extra-dex`、不是自动改库。探针只产线索，归证由人工复核录入。
（例外：`dexload-hook` dump 出的 `.dex` 文件**用** `fxapk analyze <apk> --extra-dex <dex...>` 回灌静态——那是 dex 不是 console 文本。）

## 6. 合规红线

1. **主动枚举企业号 / 打配置端点 = 主动外联，必须先取得授权。** `tenant-enum-helper.js` 的 `TENANT_LIST` 默认空，红线注释+占位符强制现场确认授权后再填；只用案卷**已知**企业号小批量验证，不做大规模盲枚举。
2. **只取端口 / 状态码 / 结构，不取正文**——对未授权的第三方端点（CDN/OSS 回源、疑似真后端）做连通性/归属判断时，只记录 IP:端口、HTTP 状态码、TLS SNI，**不抓取响应正文**。
3. **绝不假装抓到。** 探针抓不到就打"未命中+下一步"，线索栏写"未取得"，**不得编造**域名/IP/key/聊天内容。所有 `[LEAD]` 必须有原始 dump 佐证。
4. **境外目标不自取。** 真源站/IP 落在境外时，不自行访问/枚举/抓取，按程序走跨境溯源或情报协作，探针侧到"标注疑似境外 + 提供已抓到的本地侧线索"为止。

---

**一句话路线图**：`anti-detection + unpinning` 先注入护住现场 → `coldstart` 无账号抓全分发层/DNS/内嵌配置 → 标出 `[TENANT?]`/`[BACKEND!]` → `cipher/encoding/sign` 解密内嵌配置与签名 → 授权后 `tenant-enum` 小批量逼配置 → 拿到真源站则 `interceptor/websocket/webview/webview-inject/sqlite` 取物证，拿不到则 `dns + CDN/OSS` 溯源穿透分发层。线索一律手动回灌台账，红线全程不越。
